## 개요

온프레미스 DB 장애에 대비한 **백업 자동화 + Azure Storage 업로드 + Standby DB 전환** 체계를 구성한 단계. 민감 정보(DB 접속 정보, Storage Key)는 전부 **Azure Key Vault**로 분리해 관리했다.

**핵심 키워드**: Azure Key Vault · Managed Identity · Azure Storage · mysqldump · cron 자동화 · Standby DB

---

## DB 백업 자동화 구성

1. **백업 스크립트** (`/home/azureuser/backup_to_storage.sh`)를 VMSS 인스턴스 내부에 생성
2. 스크립트는 **Managed Identity**로 Azure에 로그인한 뒤, **Key Vault에서 DB 접속 정보와 Storage Account Key를 조회** — 스크립트 안에 비밀번호를 하드코딩하지 않는 구조
3. 조회한 정보로 Azure MySQL Flexible Server(`tuna-mysql-replica`)에 SSL 접속해 `mysqldump` 수행
4. 생성된 백업 파일(파일명에 DB명+시각 포함)을 Azure Storage Account(`tunatfstate604`)의 **`dbbackup` Container**로 업로드
5. `RETENTION_DAYS=7`로 설정해 7일이 지난 백업은 자동 삭제
6. `cron`에 등록해 **매일 03:00 자동 실행**

```
백업 흐름: Managed Identity 로그인 → Key Vault에서 Secret 조회
         → mysqldump 실행 → Azure Storage(dbbackup) 업로드 → 7일 지난 파일 삭제
```

> 🖼️ *사진 자리 — backup_to_storage.sh 스크립트 화면*

## Key Vault 기반 민감정보 관리

- DB 사용자명, 비밀번호, DB 이름, Storage Account Key를 **전부 Key Vault Secret**으로 등록
- 백업 스크립트는 실행 시점에 Key Vault를 조회해서 값을 가져오는 구조라, **코드/스크립트 어디에도 민감 정보가 평문으로 남지 않는다**

> 🖼️ *사진 자리 — Key Vault 기반 백업 정보 조회 설정 화면 (Secret 조회 코드 부분)*

## Standby DB 구성

- Azure Database for MySQL Flexible Server를 **Standby DB**로 구성
- 온프레미스 DB Server 장애 시, 백업 데이터 + Standby DB 전환을 통해 서비스 복구 경로 확보

> 🖼️ *사진 자리 — cron 등록 확인 화면 (매일 03:00 실행) + Standby DB 전환 테스트 결과 화면*

## 정리

- 이 단계에서 가장 신경 쓴 부분은 **"백업을 한다"가 아니라 "백업 정보 자체도 안전하게 관리한다"**는 점이었다. Key Vault + Managed Identity 조합으로 스크립트 안에 비밀번호를 남기지 않는 구조를 실제로 만들어봤다.
- cron으로 매일 새벽 자동 백업 + 7일 이후 자동 삭제까지 구성해서, 사람이 매번 신경 쓰지 않아도 되는 최소한의 운영 자동화를 경험했다.
